No. Every L1 program can be transformed into an equivalent L11 program by substituting each defined variable with its corresponding value expression throughout the program.
Since L1 does not support higher-order functions, closures, or complex binding mechanisms, define serves mainly as syntactic convenience rather than essential expressive power.
Therefore, define is not fundamentally required for expressiveness in L1.

1. 2. 
No. Every L2 program can be transformed into an equivalent L21 program.
Unlike L1, L2 includes lambda expressions, which allow variable bindings through function abstraction and application. Therefore, any use of define can be replaced by lambda expressions.
For example:
(L2
  (define x 5)
  (+ x 3))
can be transformed into:
((lambda (x) (+ x 3)) 5)
Thus, define is not essential in L2, and all L2 programs can be rewritten in L21.

1. 3. 
No. Every L2 program can be transformed into an equivalent L22 program.
The restriction to single-parameter procedures does not reduce expressive power because any multi-parameter function can be transformed using currying.
For example:
(lambda (x y) (+ x y))
can be transformed into:
(lambda (x)
  (lambda (y)
    (+ x y)))
Additionally, since the value of a procedure body is determined by its final expression, L2 procedures can generally be restructured into equivalent single-expression bodies.
Therefore, L22 remains equally expressive as L2.

1. 4. 
Yes, there are L2 programs that cannot be expressed in L23 in the same general way.
Although in some closed programs we can inline a known function argument, this does not preserve the general higher-order abstraction. L2 allows procedures to receive procedures as arguments, while L23 forbids this.
For example:
(define apply-to-5
  (lambda (f)
    (f 5)))
This procedure can receive any function f and apply it to 5. It can be used with different functions:
(apply-to-5 (lambda (x) (+ x 1)))  ==> 6
(apply-to-5 (lambda (x) (* x x)))  ==> 25
In L23, such a procedure cannot be defined, because procedures cannot receive functions as arguments. Therefore, L23 cannot express this higher-order abstraction, and the restriction reduces the expressive power of the language.

1. 5. 
Special forms are required because some language constructs need a special evaluation rule. Primitive operators are evaluated by first evaluating all their operands and then applying the operator. This evaluation strategy is not suitable for constructs such as if or define.
For example, consider:
(if (= x 0)
    1
    (/ 1 x))
If if were a primitive operator, all operands would be evaluated before applying it. Therefore, when x = 0, the expression (/ 1 x) would also be evaluated, causing a division-by-zero error. However, the correct behavior of if is to evaluate only the condition first, and then evaluate only the selected branch.
Therefore, if must be a special form and cannot be defined as a regular primitive operator.

1. 6. 
In pure functional programming, a function body with multiple expressions is not required. Since there are no side effects, the values of all expressions except the last one are discarded and do not affect the final result. Therefore, such a body can be replaced by its last expression.
Multiple-expression bodies are useful mainly in imperative or procedural languages, where expressions or statements may have side effects, such as assignment, printing, mutation, or I/O.
In L3, the syntax allows procedure bodies with one or more expression, However, since it is functional, multiple expressions are not essential for expressiveness; they are mainly a convenience, and the value of the body is the value of the last expression.

1. 7. 
A lexical address is a representation of a variable occurrence by its lexical position, rather than only by its name. It identifies the binding of the variable according to the static structure of the program: how many enclosing scopes we must go outward, and the position of the variable in that scope.
For example:
(lambda (x y)
  (lambda (z)
    (+ x z)))
Inside the inner lambda, z is bound in the current scope, so its lexical address is <0,0>. The variable x is bound in the immediately enclosing lambda, where it is the first parameter, so its lexical address is <1,0>. If y appeared there, its address would be <1,1>.
Thus the body (+ x z) can be represented as (+ <1,0> <0,0>). Lexical addresses make variable lookup depend on the lexical structure of the program.

1. 8. 
There are two possible representations for primitive operations.
In the PrimOp representation, primitive operators have a special syntactic and semantic representation, such as PrimOp("+"). An advantage of this method is that it is simple and direct to implement: the interpreter can immediately identify primitive operators and apply special built-in code for them.
In the Closure representation, primitive operators are represented like user-defined procedures. The interpreter predefines each primitive operator as a Closure in the environment, and the operator name is parsed as a VarRef. An advantage of this method is uniformity: primitive operations and user-defined procedures are treated in the same way. This also makes it more natural to pass primitive operations as values or bind them to names.


1. 9. 
a. Switching from applicative order to normal order is justified when some arguments may not be needed. Normal order delays the evaluation of arguments until their values are actually required. This can avoid unnecessary computation, errors, or non-termination.
For example:
(if #t 5 (/ 1 0))
Under normal order, the condition is evaluated first, and since it is true, only the value 5 is returned. The expression (/ 1 0) is never evaluated.
This allows the program to terminate successfully without error.
Under applicative order, all the oprands will evaluated before applying the procedure, and therefore (/ 1 0) will fail the program.
Therefore, normal order is useful when delayed evaluation can improve safety, flexibility, or termination behavior.

b. Switching from normal order to applicative order is justified when arguments are used more than once, because normal order may duplicate computations. Applicative order evaluates each argument once before applying the procedure.
For example:
((lambda (x) (+ x x)) (* 2 3))
Under applicative order, (* 2 3) is evaluated once to 6, and then (+ 6 6) gives 12. Under normal order, the expression becomes (+ (* 2 3) (* 2 3)), so (* 2 3) is evaluated twice. Therefore, applicative order can be more efficient.

1. 10. 
  a. valueToLitExp is used in the substitution-model interpreter to convert evaluated values back into literal expressions. This is needed because substitution operates on AST expressions, while evaluation of arguments produces runtime values. Therefore, before substituting an evaluated argument into the body of a procedure, the value must be represented as a literal expression.

  b. In the normal-evaluation interpreter, arguments are not evaluated before substitution. Instead, the original argument expressions are substituted directly into the body of the procedure. Since substitution receives expressions rather than evaluated values, there is no need to convert values back into literal expressions. Therefore, valueToLitExp is not needed.

  c. In the environment-model interpreter, procedure application is implemented by extending the environment with bindings from parameters to their evaluated argument values. The body of the procedure is then evaluated in this extended environment. Since no substitution into the AST is performed, values do not need to be converted back into literal expressions. Therefore, valueToLitExp is not needed.

1. 11. 
 a. Renaming is not required in the environment model because the interpreter does not perform syntactic substitution. Instead, procedure application is implemented by extending an environment with bindings from parameters to values, and variables are resolved by looking them up in the environment. Since expressions are not inserted into other expressions, there is no danger of variable capture. Lexical scope is preserved by closures, which store the environment in which a procedure was created.

 b. If the term being substituted is closed, renaming is not required for avoiding variable capture. Variable capture can only occur when a free variable in the substituted term becomes accidentally bound by a binder in the expression into which it is substituted. A closed term has no free variables, so there are no free variables that can be captured. Therefore, in this case, substitution can be performed without renaming.
